## 风险与缓解

- 如果继续把正式执行端寄托在 Unity 上，后面产品化成本会持续抬升。  
  缓解：Unity 只保留为参考客户端与潜在 sidecar。
- 如果让 Web 直接承担执行责任，会碰到浏览器权限、长连接和后台保活问题。  
  缓解：Web 只做控制端，执行统一由 `Node Agent` 承担。
- 如果没有统一的 `Node` 抽象，本地电脑和用户云服务器会变成两套产品。  
  缓解：统一节点模型、统一注册、统一控制、统一运维。
- 如果没有 session 隔离，一个节点多角色并发时容易相互污染。  
  缓解：每个角色独立 `Bot Session`，统一由 `Bot Session Manager` 调度。
- 如果 headless 只复刻 socket 协议，不复刻 Runtime GraphQL，合成、pending queue、提取和取消都会缺失。  
  缓解：把 `runtime-graphql-adapter` 作为正式架构组件，而不是临时补丁。
- 如果没有补上客户端的 token / allowance 前置语义，合成链路会和真实客户端偏离。  
  缓解：节点端在 `craft executor` 里显式处理 token 余额、`getUserGameTokenAllowance`、`getUserGameInternalToken` 以及必要的扣款授权语义。
- 如果把 AI 缩成事后优化项，架构很容易滑回规则脚本。  
  缓解：在架构层先锁定 `AI Planner / Policy` 是正式组成部分，Runtime 只负责安全执行。
- 如果实时观察只做视频流，用户会“能看不能理解”。  
  缓解：观察层必须同时输出 `Scene / Action / Decision / Economy` 四条流。
- 如果高保真预览过早自研 renderer，会背上过重客户端债务。  
  缓解：先做 `v1`、`v2`，`v3` 优先考虑 Unity sidecar 验证价值。
