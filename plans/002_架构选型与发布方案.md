## 正式数据边界

- 正式输入面 = `客户端可观测状态 + 链上/公共 API 状态`
- 允许：
  - Unity 本地状态
  - Unity 正常玩家链路拿到的客户端响应
  - 链上公开数据
  - entitlement / billing 状态
  - xlsx 静态配置
- 不允许：
  - 直接读 `game-services` 私有玩家状态
  - 直接读 `scene-server` 内部实体快照
  - 依赖任何后台独享且客户端不可见的数据

## 执行与校验边界

- `game-services-v2`
- `scene-server-v2`
- `services`

这些仓库的重要性在于：

- 业务执行
- 业务校验
- 客户端请求响应语义

但它们不属于 Bot 的直接观测层。

## Bot 决策与执行边界

- AI 只输出结构化决策。
- Executor 只接收结构化计划，不接收自由文本。
- 执行链路保留：
  - `Entitlement Check`
  - `Safety Guard`
  - `Structured Executor`
- 其中 `Safety Guard` 只在必要场景介入，例如：
  - entitlement 不满足
  - 不可逆或链上动作
  - 明显异常高成本
  - 人工接管
  - 连续异常失败
- 低风险、可重复、可验证的常规收益动作默认不应被拉入重审批流。

## 参考项目借鉴方式

参考项目：

- [minecraft-mcp-server](https://github.com/yuniko-software/minecraft-mcp-server)

借鉴重点是“工具化架构”，不是“自由对话控制”：

- `Connection Layer`
- `Operation Atom Layer`
- `State Memory`
- `Planner`
- `Executor`

在 Lumiterra 里的映射应是：

- `Connection Layer` -> Unity Runtime / 运行时状态 / 连接状态
- `Operation Atom Layer` -> `TaskAtoms / FarmingAtoms / RecoveryAtoms / WalletAtoms / NavigationAtoms`
- `State Memory` -> 最近行为、失败原因、近期收益、场景事件
- `Planner` -> 机会排序与动作序列规划
- `Executor` -> 只执行结构化计划

## 风险与缓解

- 如果继续把 `apps/aiBot` 误当成最终完整产品壳，架构会偏离权威文档。
  缓解：本仓库实现状态和最终产品框架明确分开。
- 如果 AI 越权直接控底层动作，会破坏风控和计费边界。
  缓解：统一采用 `Entitlement Check + Safety Guard + Structured Executor`。
- 如果把所有动作都放进强 guard 流，系统会退化成低效脚本而不是收益优化机器人。
  缓解：把 `Safety Guard` 定义成必要介入层，而不是默认全量拦截层。
- 如果继续用旧的 Unity 路径和旧的技术框架描述，后续实现会反复返工。
  缓解：所有技术框架统一回到 `docs/lumiterra-aibot-product-plan.md`。
