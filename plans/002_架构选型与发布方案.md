## 风险与缓解

- 如果继续把 `Unity client` 当正式执行内核，后续发布、多开、托管和扩展都会越来越重。  
  缓解：正式执行统一走 headless runtime，`Unity` 只做参考和候选 sidecar。
- 如果 `v1` 一开始就按大众轻用户设计，架构会为了低门槛兼容牺牲交付速度。  
  缓解：`v1` 先聚焦可安装 `Node Agent` 的重度用户，后面再扩展更低门槛形态。
- 如果让 `Web` 直接承担执行责任，会在权限、保活、跨设备控制上持续受限。  
  缓解：严格坚持“`Web` 控制、`Node` 执行”的分离模型。
- 如果只复刻 gateway，不复刻 Runtime GraphQL，合成等关键链路会缺失。  
  缓解：把 `runtime-graphql-adapter` 定义为正式架构组件。
- 如果不补齐 allowance、internal token、pending queue 等语义，合成链路会与真实客户端偏离。  
  缓解：把这些语义列为 Runtime GraphQL 最小兼容集。
- 如果状态分别挂在 gateway 和 GraphQL 两边，没有统一 `world-model`，后面会长期分裂。  
  缓解：所有状态最终归一到统一的 session world state。
- 如果 AI 直接下发低层协议动作，执行风险和不稳定性会快速放大。  
  缓解：AI 只在受限动作层决策，Runtime 保持最终执行控制权。
- 如果没有节点升级治理，协议和 schema 演进后节点会迅速碎片化。  
  缓解：从架构层就引入版本、能力、通道、manifest 和回滚要求。
