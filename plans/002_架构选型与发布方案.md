## 风险与缓解

- 如果把完整 AI 主决策、模型 key 和成本治理都塞进 `Unity Client`，官方成本、额度和审计都会失控。  
  缓解：主 AI 决策留在平台侧 `ai-brain-service`，模型调用统一走 `ai-gateway`。
- 如果把完整 AI 主决策全部放到平台侧而没有客户端战术层，地图、寻路、周围怪物/NPC、场景态和接管体验会明显变差。  
  缓解：在客户端内保留 `local-world-model + node-tactical-policy + runtime-guard`。
- 如果多个角色共享同一份记忆、路线权重或进化状态，最终会收敛成同质化 Bot，并产生上下文污染。  
  缓解：按角色强隔离 `role-brain-session + memory-store + policy-state-store + pnl-ledger + decision-journal`。
- 如果把 `赚钱型 / 成长型 / 平衡型` 实现成固定静态模式，角色在不同成长阶段会持续使用错误策略。  
  缓解：在架构层把三者只定义为初始模板，长期运行依赖 `policy-state-store` 中可演化的阶段性策略状态。
- 如果 `Unity` 仍要求用户手工录入复杂绑定信息，用户路径会变重且容易出错。  
  缓解：由 `Unity` 按当前钱包自动查询 `AIBot(机器人)`，空状态下只保留“去 `Web` 创建”和“重新加载”。
- 如果没有正式的单活会话锁和动态通信标识，同一 `AIBot(机器人)` 可能被多个客户端同时滥用。  
  缓解：引入 `session-lease-manager`，按启动会话签发动态通信标识并用心跳维持单活租约。
- 如果客户端和平台侧之间没有正式的长连接会话层，实时文字流、审批、状态推送和接管反馈会不稳定。  
  缓解：正式引入 `Agent Session Channel (WSS)`。
- 如果决策请求没有结构化 RPC 边界，模型上下文会膨胀，计量与审计也会失真。  
  缓解：正式引入 `Decision RPC (HTTPS)`，并只传结构化上下文。
- 如果只兼容 game channel，不兼容 Runtime GraphQL，合成链路会缺失。  
  缓解：把 `runtime-graphql-adapter` 定义为客户端内正式适配层。
- 如果没有 `economy-chain-adapter`，财富值绑定/熔炼、宝箱上链和出售链路会断裂。  
  缓解：把链上经济动作单独抽成正式适配层，而不是混进普通 game channel。
- 如果没有正式的盈亏记录与决策日志，AI 的“自我进化”会退化成不可解释的黑箱漂移。  
  缓解：把 `decision-journal + pnl-ledger + route-evaluator` 作为正式架构组件。
- 如果 AI 费用无法精确计量和对账，当前收费模式就无法成立。  
  缓解：从架构层明确引入 AI 用量计量、成本归集和账单对齐能力。
- 如果客户端代码更新和策略参数更新绑在一起，后续迭代会非常重。  
  缓解：客户端代码与资源更新走客户端发布治理，提示词、权重和公共情报走平台侧远程配置治理。

- 上文描述的是**完整产品**的组件边界（Unity、平台侧 `ai-brain-service`、`ai-gateway`、会话租约等）。
- **当前本仓库**主要承载 **`Web AIBot Management` 的最小实现**：
  - 路径：`apps/aiBot`（`Next.js` + `pages/api` + `PostgreSQL`）
  - 前端：`wagmi` 连接钱包；**无**平台侧 `AuthSession` / 钱包验签会话；`create/update` API **信任请求体中的 `walletAddress`**（仅适合开发或后续加鉴权）。
  - 数据：单表 `aibots`（一钱包一行）。
  - 详情见 `plans/005`。
- `apps/aiBrainService` 仍为**独立占位应用**，与 `apps/aiBot` 内已删除的空目录无关。
