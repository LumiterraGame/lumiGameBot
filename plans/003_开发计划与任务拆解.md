## 目标

基于 `001` 的总体规划和 `002` 的架构结论，给出只面向交付的开发步骤拆解、里程碑、同步标准、验收标准和对应依赖。这个文件不再重复讨论产品定位和架构取舍，只回答“先做什么、后做什么、每一步产出什么、依赖什么、怎么验收”。

## 范围内

- 只覆盖 `Phase 0` 和 `Phase 1`。
- 只覆盖 `M0` 到 `M5` 的详细开发步骤。
- 明确每个里程碑的目标、交付物、依赖和验收口径。
- 明确每个里程碑结束时必须同步的文档和状态。
- 明确首个开发主线是：
  - `Web` 端配置入口
  - `Node Agent`

- 明确后续增量主线是：
  - 最小可观察
  - AI 受限动作决策

  - “合成业务”第一条收益闭环

- 明确 `v1` 的 AI 接入要求：
  - 尽量兼容主流 LLM
  - 至少不绑死单一模型供应商
  - 第一版实现允许先验证 `GPT / Claude`，但接口层要保留兼容空间

## 任务拆解（按顺序执行）

1. `M0`：先完成 `Web` 端最小配置入口。  
   交付物：
   - Bot 基础配置页
   - 钱包/账号配置入口
   - 默认风险策略配置展示为“高风险动作需审批”
   - 默认运行模式展示为“观察为主，可接管”
   - 节点接入引导
   - 启动/停止按钮
   - session 基础状态页

2. `M0`：补齐控制面最小能力，支撑 `Web` 先落地。  
   交付物：
   - 配置存储
   - session 创建接口
   - 启动/停止接口
   - 节点状态查询接口

3. `M0` 验收。  
   验收口径：
   - 用户可在网页完成基础配置
   - 配置可保存、可读取、可触发启动流程

4. `M1`：实现 `Node Agent` 最小骨架。  
   交付物：
   - 节点注册
   - 心跳
   - 版本上报
   - 能力上报
   - 拉取配置
   - 接收启动/停止/暂停命令

5. `M1`：打通 `Web / Control Plane / Node` 的最小闭环。  
   交付物：
   - 网页能看到节点在线状态
   - 控制面能把配置和命令下发到节点
   - 节点能回报命令执行结果

6. `M1` 验收。  
   验收口径：
   - 一个节点可被注册、识别、控制
   - 网页的启动/停止操作能真正作用到节点

7. `M2`：打通登录与身份链路。  
   交付物：
   - `loginByMetamask`
   - `authByMetamask`
   - `loginByTemporaryToken`
   - `profile`
   - `getUserWallets`
   - `web3Profile`
   - `auth-adapter`
   - `identity-adapter`

8. `M2`：实现 `gateway-client-adapter` 最小可用版本。  
   交付物：
   - game channel 登录
   - 基础玩家态查询
   - 进入场景所需最小动作链
   - 基础心跳
   - 基础断线重连

9. `M2`：建立单 Bot session 的最小生命周期。  
   交付物：
   - session 创建
   - session 启动
   - session 停止
   - session 异常态记录

10. `M2` 验收。  
    验收口径：
    - 用户在网页完成配置
    - 节点成功启动单 Bot
    - Bot 成功登录
    - Bot 成功进入有效游戏场景
    - 异常断线可基本恢复

11. `M3`：建立最小 `world-model`。  
    交付物：
    - 登录态
    - 节点态
    - session 态
    - 玩家基础态
    - 场景/坐标态
    - 当前执行步骤

12. `M3`：把最小运行态回传到网页。  
    交付物：
    - 节点在线状态

    - 最近动作
    - 最近错误

13. `M3` 验收。

## 风险与缓解

- 如果在 `M2` 之前就同时推进 AI、GraphQL、收益闭环，主线会失焦。  
  缓解：先把“网页 -> 节点 -> Bot 进游戏”打成首个硬闭环。
- 如果 `M0` 没有先把配置入口做出来，后续验证会长期依赖手工改配置。  
  缓解：优先交付 `Web` 配置入口。
- 如果 `M1` 的节点骨架不稳定，后面的所有运行能力都会失去统一承载点。  
  缓解：在进入游戏前先稳定节点注册、心跳和命令链路。
- 如果 `M2` 不能稳定进入游戏，后续所有收益和 AI 验证都不可信。  
  缓解：把“Bot 稳定进入游戏”作为首个硬里程碑。
- 如果 `M4` 没有受限动作集，AI 决策会直接放大执行风险。  
  缓解：先定义 `action DSL`，再接入 AI。
- 如果 `v1` 把 AI 接入层写死到单一供应商，后面切模型和并行验证成本会很高。  
  缓解：`v1` 就引入 provider 抽象，至少保证对主流 LLM 的兼容空间。
- 如果 `M5` 的首条收益闭环选得太大，节奏会被拖垮。  
  缓解：已收敛为“合成业务”单条闭环，不再在 `v1` 并行比较多条路线。
- 如果 `v1` 不把高风险动作默认收进审批流，首版信任成本会过高。  
  缓解：默认以观察为主，高风险动作必须人工审批。
- 如果每个里程碑没有同步标准，后续状态会混乱且难以接续。  
  缓解：每个里程碑结束必须同步交付物、依赖、验收和阻塞项。
